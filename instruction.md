# Task Proposal: Git DAG Reconstruction from Obfuscated Packfiles

## 1. Why is this task genuinely difficult?
* **Core Challenge:** During a VCS migration, a repository's `.pack` file was written with custom delta offset obfuscation (each delta base offset in OBJ_OFS_DELTA objects is bitwise XORed with a constant pattern) and a zeroed-out checksum trailer. Because of this custom byte-level obfuscation, standard Git plumbing commands (like `git index-pack` and `git unpack-objects`) abort immediately with corruption errors. The solver must manually parse the binary packfile at the byte level, decode the variable-length header bytes, de-obfuscate the delta offsets, resolve the zlib object streams, and output a valid reconstructed history DAG.
* **Professional Role:** Senior DevOps / Infrastructure Engineers, Git Specialists, or Systems Engineers who perform custom VCS migrations, repository forensics, and low-level data recovery.
* **Data Complexity:** The data is fully synthetic, generated by a fixed script containing a history of 15 commits, including 3 branches, 2 merges (one standard 2-parent merge and one 3-parent octopus merge), and 2 file renames. The pre-corruption ground truth is captured by serializing the original Git repository DAG structure directly to a read-only verifier path.
* **Pitfalls & Cruxes:**
  1. **Defeating Native Tooling:** Because the packfile is intentionally corrupted/obfuscated, native Git commands fail. The agent must implement a custom packfile reader to decode variable-length integers, de-obfuscate offsets, and manually resolve delta base chains (OBJ_OFS_DELTA).
  2. **Disambiguating Trees & Blobs:** Distinguishing between actual content updates, directory structure changes (tree objects), and commit metadata without relying on built-in high-level index files.
  3. **DAG Resolution:** Reconstructing the commit tree and identifying branch heads by finding commits that have no children in the parent-pointer references.

## 2. Intended Solution Approach
* **High-Level Strategy:**
  1. Parse the `.pack` binary stream, skipping the standard 12-byte header to read each object entry.
  2. Decode the variable-length size and type headers. For OBJ_OFS_DELTA type objects, decode the negative offset and apply the de-obfuscation key (XOR 0x5A) to recover the true base object offset.
  3. Decompress object payloads using zlib. Reconstruct delta objects by applying diff hunks to their base objects.
  4. Parse commit headers to extract `parent` commits, `tree` hashes, `author` names/emails, and `author date` timestamps.
  5. Reconstruct the DAG topology and serialize the results to `dag_reconstructed.json` in `/app/`.
* **Expert Effort Estimate:** ~3–4 hours for a domain expert who understands Git binary layouts; ~12–16 hours to write a robust, from-scratch parser from the ground up.

## 3. How will the solution be verified?
* **Agent Outputs:** The agent must output a `dag_reconstructed.json` file in `/app/` and recreate the correct loose object store in `.git/objects/` (or rebuild a valid index) so that standard `git checkout` works.
* **Verification Criteria:**
  1. **DAG Schema and Matching:** The verifier compares `dag_reconstructed.json` (keys are 40-character lowercase hex commit hashes, values contain `parents` (list of parent SHAs, sorted alphabetically), `tree` (tree hash), `author` (normalized as `"Name <email>"`), `date` (author Unix timestamp), and `message` (commit message string, trimmed of trailing newlines)). It asserts an exact dictionary match.
  2. **Checkout Correctness:** The verifier checks out each identified leaf commit SHA (e.g. `git checkout <leaf-sha>`) and verifies the working-tree files are byte-identical to the original uncorrupted states.
  3. **Object Signatures:** Verification of the SHA-1 signatures of reconstructed objects to ensure no metadata was mutated during recovery.
